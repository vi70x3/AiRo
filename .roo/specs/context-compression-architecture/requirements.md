# Requirements Document

## Introduction

The Context Compression Architecture introduces a layered retrieval model for system prompt and environment context injection. Currently, the agent receives the full system prompt (identity, safety, tool docs, spec docs, workspace listing, rules, MCP catalog, todo list) every turn, consuming 15k-40k tokens per turn. This feature replaces the monolithic injection model with four retrieval layers, each loaded conditionally based on token budget and current agent state. The target is 2k-6k tokens per turn while preserving task completion quality.

This phase builds on Phase 1 (Semantic Loop Detection) by adding structured evidence preservation to the condensation system and introducing tiered environment details. It integrates with the existing `generatePrompt()` function in [system.ts](src/core/prompts/system.ts:44), the existing `getEnvironmentDetails()` function in [getEnvironmentDetails.ts](src/core/environment/getEnvironmentDetails.ts:23), and the existing condensation pipeline in [index.ts](src/core/condense/index.ts:254).

## Glossary

- **Agent**: An AI-powered assistant that processes user requests and executes tasks using available tools and reasoning capabilities.
- **Layer 0 (Permanent Core)**: The minimal set of instructions that must be present in every system prompt: identity, safety rules, tool syntax, and execution rules. Budget: 1500 tokens maximum.
- **Layer 1 (Active Mode)**: Mode-specific instructions loaded dynamically only when the agent is in a mode that requires them (e.g., debug mode diagnosis steps, spec mode workflow rules).
- **Layer 2 (Active Spec)**: A compressed JSON summary of the current spec's requirements, design, and tasks, replacing full document injection.
- **Layer 3 (Symbol Retrieval)**: Symbol-level retrieval via `search_symbols()` that returns only the symbols needed for the current task, replacing full file reads.
- **Context Budget**: The target token count per turn for system prompt plus environment details. Current: 15k-40k. Target: 2k-6k.
- **Token Counter**: The existing `estimateTokenCount()` function in [context-management/index.ts](src/core/context-management/index.ts:33) used to measure token counts for content blocks.
- **Environment Details**: The auto-generated context block produced by `getEnvironmentDetails()` in [getEnvironmentDetails.ts](src/core/environment/getEnvironmentDetails.ts:23), injected into every user message.
- **Condensation**: The process of summarizing conversation history via `summarizeConversation()` in [condense/index.ts](src/core/condense/index.ts:254).
- **Structured Summary**: A condensation output that preserves evidence (file paths, tool names, conclusion identifiers) in a machine-readable format alongside the narrative summary.
- **Tiered Environment**: A model where environment details are split into critical information (always included) and detailed information (included only when token budget allows).
- **Spec Summary**: A compressed JSON representation of an active spec, containing spec name, one-line summary, affected files, and current phase progress.

## Requirements

### Requirement 1: Layer 0 — Permanent Core

**User Story:** As a user, I want the agent to always have the essential instructions it needs to operate safely and correctly, regardless of what other context is loaded.

#### Acceptance Criteria

1. THE System_Prompt_Generator SHALL define a Layer 0 (Permanent Core) containing: agent identity (role definition), safety rules, tool syntax (from `getSharedToolUseSection`), execution rules (from `getToolUseGuidelinesSection`), and markdown formatting rules (from `markdownFormattingSection`).
2. THE Layer 0 content SHALL not exceed 1500 tokens as measured by `estimateTokenCount()`.
3. WHEN the token count of Layer 0 exceeds 1500 tokens, THE System_Prompt_Generator SHALL log a warning identifying the specific sections that caused the overflow.
4. THE Layer 0 content SHALL be included in every system prompt, regardless of mode, configuration, or token budget state.
5. THE Layer 0 content SHALL be generated by a new `buildLayer0Core()` function that accepts the same `toolContext` parameter as the existing section functions.
6. IF a mode's `roleDefinition` causes Layer 0 to exceed 1500 tokens even without additional sections, THEN THE System_Prompt_Generator SHALL truncate the roleDefinition to the first paragraph and append a note indicating truncation.

### Requirement 2: Layer 1 — Active Mode Instructions

**User Story:** As a user, I want mode-specific instructions loaded only when needed so that context budget is not wasted on instructions irrelevant to the current mode.

#### Acceptance Criteria

1. THE System_Prompt_Generator SHALL define a Layer 1 (Active Mode) containing: mode-specific instructions from `addCustomInstructions()` and mode-specific rules from `getRulesSection()`.
2. WHEN the agent is in "vibe" mode (the default execution mode), THE System_Prompt_Generator SHALL include only the base mode instructions and SHALL omit extended workflow rules that are specific to other modes.
3. WHEN the agent is in "debug" mode, THE System_Prompt_Generator SHALL include debug-specific instructions: diagnose, log, confirm, and fix workflow steps.
4. WHEN the agent is in "spec" mode, THE System_Prompt_Generator SHALL include spec workflow rules and SHALL reference the compressed spec summary from Layer 2.
5. THE Layer 1 content SHALL be loaded dynamically based on the current mode slug, not statically concatenated.
6. THE System_Prompt_Generator SHALL use a mode-to-instruction mapping that is extensible: adding a new mode requires only adding a new entry to the mapping, not modifying the prompt generation logic.
7. WHEN a mode is not found in the mode-to-instruction mapping, THE System_Prompt_Generator SHALL fall back to the base mode instructions and log a warning.

### Requirement 3: Layer 2 — Active Spec Summary

**User Story:** As a user, I want the agent to have access to the current spec's essential information without injecting the full spec documents into the context.

#### Acceptance Criteria

1. THE System_Prompt_Generator SHALL define a Layer 2 (Active Spec) containing: a compressed JSON summary of the active spec, when a spec is active.
2. THE compressed spec summary SHALL contain at minimum: `spec` (kebab-case name), `summary` (one-line description), `affectedFiles` (array of file paths), and `currentPhase` (current phase name with progress status).
3. WHEN no spec is active, THE System_Prompt_Generator SHALL omit Layer 2 entirely, producing zero tokens for this layer.
4. THE compressed spec summary SHALL be generated by a `buildSpecSummary()` function that reads the active spec's directory in `.roo/specs/<spec-name>/` and produces the JSON structure.
5. THE `buildSpecSummary()` function SHALL read only the first 20 lines of each spec document (requirements.md, design.md, tasks.md) to extract summaries, and SHALL NOT inject the full document content.
6. THE compressed spec summary SHALL not exceed 500 tokens as measured by `estimateTokenCount()`.
7. IF the spec directory does not exist or is empty, THEN THE `buildSpecSummary()` function SHALL return an empty string and log a warning.

### Requirement 4: Layer 3 — Symbol Retrieval Integration

**User Story:** As a user, I want the agent to retrieve only the symbols it needs rather than reading entire files, so that context is used efficiently.

#### Acceptance Criteria

1. THE System_Prompt_Generator SHALL define a Layer 3 (Symbol Retrieval) that provides symbol-level context when the repository is indexed via jcodemunch.
2. WHEN the repository is indexed, THE System_Prompt_Generator SHALL use `search_symbols()` to retrieve relevant symbols based on the current task context, rather than injecting full file listings.
3. THE symbol retrieval SHALL return only the symbols that match the current task's scope: function names, class names, and their signatures and summaries.
4. WHEN the repository is not indexed, THE System_Prompt_Generator SHALL fall back to the existing `list_files` output in environment details.
5. THE Layer 3 content SHALL be generated by a `buildSymbolContext()` function that accepts the current task description and returns relevant symbols.
6. THE `buildSymbolContext()` function SHALL not exceed 1000 tokens as measured by `estimateTokenCount()`.
7. IF `search_symbols()` returns no relevant results, THEN THE `buildSymbolContext()` function SHALL return an empty string and the system SHALL fall back to Layer 0 + Layer 1 + environment details only.

### Requirement 5: Tiered Environment Details

**User Story:** As a user, I want the agent to always have critical environment information (mode, model, current workspace) while detailed information (full file listings, terminal output, git status) is included only when the token budget allows.

#### Acceptance Criteria

1. THE Environment_Details_Generator SHALL split environment details into two tiers: Tier A (Critical) and Tier B (Detailed).
2. THE Tier A (Critical) SHALL include: current mode slug and name, current model ID, current workspace directory path, and todo list (if enabled). Tier A SHALL not exceed 500 tokens.
3. THE Tier B (Detailed) SHALL include: VSCode visible files, VSCode open tabs, actively running terminals, inactive terminals with completed process output, recently modified files, current time with timezone, git status, current cost, and full workspace directory file listing.
4. WHEN the total token budget for environment details exceeds the remaining budget after system prompt layers, THE Environment_Details_Generator SHALL include only Tier A and omit Tier B.
5. THE Environment_Details_Generator SHALL calculate the remaining budget as: `targetBudget - layer0Tokens - layer1Tokens - layer2Tokens - layer3Tokens`.
6. WHEN the target budget is not reached after including all tiers, THE Environment_Details_Generator SHALL include Tier B content in priority order: (1) recently modified files, (2) VSCode visible files, (3) actively running terminals, (4) current time, (5) git status, (6) current cost, (7) VSCode open tabs, (8) inactive terminals, (9) full workspace directory listing.
7. THE Environment_Details_Generator SHALL use the existing `getEnvironmentDetails()` function signature to maintain backward compatibility, adding an optional `tier` parameter that accepts `"critical" | "detailed" | "full"`.
8. WHEN the `tier` parameter is `"critical"`, THE Environment_Details_Generator SHALL return only Tier A content.
9. WHEN the `tier` parameter is `"detailed"`, THE Environment_Details_Generator SHALL return Tier A plus as much of Tier B as the token budget allows.
10. WHEN the `tier` parameter is `"full"`, THE Environment_Details_Generator SHALL return all environment details (current behavior, for backward compatibility).

### Requirement 6: Token Budget Enforcement

**User Story:** As a system administrator, I want the system to enforce a configurable token budget per turn so that context costs are predictable and controlled.

#### Acceptance Criteria

1. THE Token_Budget_Enforcer SHALL define a configurable `contextBudget` parameter with a valid range of 1000 to 50000 tokens and a default of 6000 tokens.
2. THE Token_Budget_Enforcer SHALL calculate the total per-turn token cost as: `layer0Tokens + layer1Tokens + layer2Tokens + layer3Tokens + environmentTierATokens + environmentTierBTokens`.
3. WHEN the total per-turn cost exceeds the `contextBudget`, THE Token_Budget_Enforcer SHALL reduce layers in the following order: (1) Layer 3 (Symbol Retrieval), (2) Layer 2 (Active Spec), (3) Layer 1 extended instructions (keeping base mode instructions), (4) Tier B environment details, (5) Layer 1 base instructions (last resort).
4. THE Token_Budget_Enforcer SHALL never reduce Layer 0 (Permanent Core) below its full content.
5. THE Token_Budget_Enforcer SHALL log each layer reduction event with: timestamp, layer removed, token count before reduction, and token count after reduction.
6. THE Token_Budget_Enforcer SHALL expose the current per-turn token count via the existing metrics system for observability.
7. WHEN `contextBudget` is set to 0, THE Token_Budget_Enforcer SHALL disable budget enforcement entirely and use the existing full-injection behavior.

### Requirement 7: Enhanced Condensation with Evidence Preservation

**User Story:** As a user, I want the condensation process to preserve structured evidence (file paths, tool names, conclusion identifiers) so that the agent can continue work accurately after context compression.

#### Acceptance Criteria

1. THE Condensation_Engine SHALL produce structured summaries that include, in addition to the narrative summary: a list of files modified (with paths), a list of tools invoked (with names and key parameters), a list of conclusions reached, and a list of active blockers.
2. THE structured evidence SHALL be formatted as a JSON block within the summary content, separated from the narrative summary.
3. THE Condensation_Engine SHALL extract evidence from the conversation history by analyzing tool_use and tool_result blocks before they are converted to text via `transformMessagesForCondensing()`.
4. THE evidence extraction SHALL occur before the LLM summarization call, not after, so that the evidence is derived from the actual tool interactions rather than the LLM's summary of them.
5. WHEN evidence extraction fails (e.g., malformed tool blocks), THE Condensation_Engine SHALL proceed with the narrative summary only and log a warning.
6. THE structured evidence block SHALL not exceed 1000 tokens as measured by `estimateTokenCount()`.
7. THE Condensation_Engine SHALL preserve the existing `getMessagesSinceLastSummary()`, `getEffectiveApiHistory()`, and `cleanupAfterTruncation()` functions without modification to their core logic.

### Requirement 8: Configuration

**User Story:** As a system administrator, I want configurable context compression parameters so that I can tune the feature for different models, use cases, and cost constraints.

#### Acceptance Criteria

1. THE Configuration_Manager SHALL add a `contextCompression` section to the `globalSettingsSchema` in [global-settings.ts](packages/types/src/global-settings.ts:118) following the existing pattern used by `loopDetection`.
2. THE `contextCompression` section SHALL contain: `enabled` (boolean, default: false), `contextBudget` (number, range: 1000-50000, default: 6000), `layer0MaxTokens` (number, range: 500-3000, default: 1500), `layer2MaxTokens` (number, range: 100-2000, default: 500), `layer3MaxTokens` (number, range: 200-3000, default: 1000), `environmentTierAMaxTokens` (number, range: 100-2000, default: 500), `enableTieredEnvironment` (boolean, default: true), and `enableEnhancedCondensation` (boolean, default: true).
3. WHEN `contextCompression.enabled` is false, THE System_Prompt_Generator SHALL use the existing full-injection behavior from `generatePrompt()` with no changes.
4. WHEN configuration values are not explicitly set, THE Configuration_Manager SHALL use documented default values.
5. THE Configuration_Manager SHALL validate each parameter independently and accept valid parameters while rejecting only invalid parameters with an error log containing the parameter name and invalid value.
6. THE Configuration_Manager SHALL apply runtime configuration updates within 1 second without requiring agent restart, while preserving the previous configuration during the update process.

### Requirement 9: Observability

**User Story:** As a system administrator, I want visibility into context compression behavior so that I can monitor effectiveness, tune budgets, and troubleshoot issues.

#### Acceptance Criteria

1. THE Observability_Logger SHALL log the following events with timestamp and relevant numeric values: layer reduction (including layer name, tokens before, tokens after), budget enforcement triggered (including total tokens and budget), tiered environment truncation (including tier B items omitted), spec summary generated (including spec name and token count), and evidence preservation (including evidence token count).
2. THE Observability_Logger SHALL provide metrics including: average tokens per turn (over last 10 turns), layer reduction frequency per session, tier B omission rate, and condensation evidence preservation rate.
3. THE Observability_Logger SHALL expose metrics as structured key-value entries containing a timestamp, metric name, and numeric value.
4. THE Observability_Logger SHALL exclude the following from all log entries: user message text, file contents, tool parameters containing user input, and reasoning content that includes task-specific information.

## Out of Scope

- **Model-specific prompt optimization**: Optimizing prompt content for specific model providers (e.g., Claude vs GPT) is deferred to a future phase.
- **Dynamic tool loading based on task**: Loading tool definitions dynamically based on the current task context is deferred to the Tools SSOT spec.
- **Cross-session context persistence**: Persisting context layers across agent sessions for continuity is deferred to a future phase.
- **MCP catalog compression**: Compressing the MCP server catalog in the capabilities section is deferred to a future phase.
- **Speculative execution**: Pre-fetching context layers before they are needed based on predicted agent behavior is deferred to a future phase.
